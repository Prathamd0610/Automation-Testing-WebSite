Administrator Guide

Admin Panel — User Guide

Automation Testing Practice Platform · Manage users, roles, account balances, products, audit history, and broadcasts.

1. Overview

The Admin Panel is a dedicated, role-protected area for platform administrators. It is only visible and accessible to users with the admin role. It centralizes six tools:

📊 Dashboard
Live counts of users, catalog, business records, and total balances.
👥 Users & Roles
Create users, switch roles, activate/deactivate, reset passwords, revoke sessions, delete.
💳 Accounts
View every user's checking/savings and credit or debit their balances.
📦 Products
Create, edit, and remove catalog products.
📜 Audit Log
A record of every sensitive admin action.
📣 Broadcast
Send a notification to all users in real time.

2. Accessing the panel

  1. Open the site and sign in with an administrator account.
  2. An “Admin Panel” section appears in the left sidebar (only admins see it).
  3. Click any item — or go directly to /admin.
Demo administrator: admin@practice.dev / Admin123!
Standard demo user (non-admin): user@practice.dev / User1234!

Non-admins never see the Admin nav, and visiting /admin directly redirects them to the dashboard.

Adjusting a balance

  1. Click Adjust on a user's row.
  2. Choose the account — Checking or Savings.
  3. Choose Credit (add) or Debit (remove).
  4. Enter an amount and an optional description (e.g. “Quarterly bonus”).
  5. Click Apply adjustment.
Every adjustment writes a transaction record and an audit entry. The change appears immediately on the user's own Banking page. A debit cannot overdraw a balance — the server returns an error if the amount exceeds the funds.

9. Security model

  • Role-gated everywhere: all admin endpoints require a valid login and the admin role (server-enforced).
  • Hidden from non-admins: the nav is conditional, and admin routes redirect non-admins away.
  • Self-protection is server-authoritative: the UI disables self-actions, and the API independently blocks them (HTTP 403).
  • Validated input: all requests are schema-validated; debits can't overdraw balances.
  • Auditable: sensitive actions are logged with actor, IP, and details.
